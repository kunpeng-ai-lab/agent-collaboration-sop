# Phase Continuity Rules

ACS projects must treat the approved phase plan as a controlled artifact.

## Principle

Do not discard, skip, rename, or reframe an approved plan because of a later chat
message. New information can trigger a plan review, but changing the roadmap is
an Owner decision.

## Required Check Before Next-Phase Planning

Before proposing the next task or next milestone, Reviewer and Executor must
check:

- current approved phase name
- current phase exit criteria
- unfinished tasks in the current phase
- open review findings
- closeout gaps
- Owner decision items
- previously approved next-phase sequence
- whether the new proposal is a continuation, correction, or scope change

## Conflict Handling

If a new request appears to conflict with the existing phase sequence:

1. State the existing approved plan.
2. State the proposed change.
3. Explain the risk of changing direction.
4. Ask Owner for explicit approval before instructing the Executor.

## Reviewer Gate

Reviewer must mark a next-phase recommendation incomplete when it does not
include a phase-continuity check.

## Executor Gate

Executor must not start a phase that was skipped into by implication. If the
instruction is ambiguous, Executor reports blocked and asks Reviewer to confirm
whether Owner approved the phase change.

## Redline

```text
No phase-continuity check -> next-phase recommendation incomplete.
Plan change without Owner approval -> scope drift.
Latest chat message does not automatically override the approved roadmap.
```
